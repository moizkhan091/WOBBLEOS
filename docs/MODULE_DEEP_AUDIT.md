# WOBBLE OS — Deep Per-Module Audit

_Generated 2026-07-10 by a 23-agent deep audit (one agent per module, reading real code). 23 modules profiled · 6 critical · 57 high · 77 medium · 59 low._

> NOTE: agent-registry and intelligence-loop agents hit a retry cap and are pending a re-run.

### memory-brain

**Purpose:** Long-term memory / Brain for WOBBLE OS: a tiered, bank-partitioned semantic memory store with a founder approval gate. Founders and upstream agents propose durable facts; the module routes them into named memory banks, embeds them for pgvector cosine retrieval, and enforces human-in-the-loop before anything becomes trusted memory. Also provides direct CRUD (add/edit/pin/archive/restore), version history, dedup + conflict detection on write, staleness review, merge/split/bulk ops, per-founder export, and a soft-delete + purge lifecycle. retrieveMemoryContext is the read contract consumed by Ask WOBBLE and workers.

**Data tables:** memory_banks, memory_records, memory_chunks, memory_bank_links, memory_conflicts, memory_record_versions, memory_update_proposals, conversations, conversation_messages, approvals, audit_logs, model_runs  
**External deps:** openrouter  
**LLM calls per run:** 1

**Agents / LLMs:**

- **query/content embedder (openai text-embedding-3-small)** — Turn query text (retrieve) or memory content (create/edit/harvest) into a 1536-dim vector for pgvector cosine search and dedup/conflict nearest-neighbour lookup. One call per retrieve, one per record create, one per chunk on edit; content embedded once and reused for both dedup search and the stored chunk. _(role: embeddings via OpenRouter (not a runTextProvider role), retrieves-first: false)_
- **memory_router (Memory Harvester extractor)** — Read a founder/WOBBLE conversation transcript and emit a JSON array of durable memory candidates. Founder-scoped candidates auto-save; shared candidates are proposed for approval. Only text-LLM in the module and it is NOT wired to any worker or route. _(role: memory_router (text), ~600 tok, retrieves-first: false)_
- **routeMemoryPlacement / suggestMemoryBanks** — Suggest which banks a proposal belongs to. Despite the router framing this is pure deterministic heuristic (sourceType hint tables + regex keyword matching), not an LLM call. _(role: none (heuristic), retrieves-first: false)_

**Data flow:** Entry: (1) POST /api/memory/proposals to proposeMemoryUpdate heuristically routes banks, writes memory_update_proposals plus an approvals row; approveMemoryUpdate embeds content and writes memory_records + memory_chunks + memory_bank_links. - **content_copywriter (self-critique/revise)** — Rigorous self-critique pass: list issues and output a stronger revised draft. On unparseable output it silently falls back to the original draft (still billed). _(role: content_copywriting, ~1600 tok, retrieves-first: false)_

This LLM self-score alone drives the packet's quality gate (passesQualityGate = postWorthiness pass AND all six dims >=7). _(role: content_scoring (default openai/gpt-4o-mini — cheap tier), ~1600 tok, retrieves-first: false)_

**Data flow:** Enters via POST /api/content/graph (requireFounder; actor from session, never body) -> enqueues a content.graph job on the 'general' queue (maxAttempts 2). Worker registry dispatches runContentGraphJobHandler -> runContentGraph. Node 1 retrieves in parallel: core memory brain, knowledge for the objective, and the approved-intelligence context block; runs strategy. **Flaws (8):**

- 🟠 **cost/token** — No checkpointing: any of nodes 1/2/3/5 returning unparseable JSON throws and fails the whole job, discarding all prior successful (and already-billed) LLM calls. - 🟡 **security** — Prompt injection via ingested content: source chunks and knowledge notes (which can originate from Apify-scraped competitor pages / untrusted ingested sources) are embedded verbatim into the researcher prompt with no delimiting or instruction-hardening beyond 'never invent'. The researcher's output then drives the copywriter and a founder-facing packet. _Fix:_ Wrap untrusted note/chunk text in explicit fenced delimiters, add a system instruction that content inside is data not instructions, and strip/neutralize injection patterns; consider a provenance check that drops points citing indexes the model didn't actually receive.
- 🟡 **correctness** — The graph's quality gate trusts the scoring LLM's own self-report only: passesQualityGate(score.selfReview) with no objective checks. The cheaper single-call path (content-worker) uses an OBJECTIVE gradeContentExcellence gate, but the multi-agent graph does NOT — the two paths gate inconsistently and the 'premium' path is the weaker gate. _Fix:_ Run gradeContentExcellence (the objective gate used by content-worker) on the final copy in the graph too, and require BOTH the objective gate and the LLM self-score to pass before requesting approval.
- 🟡 **cost/token** — No idempotency or concurrency guard on the graph run. The route never derives an idempotencyKey (it's optional and the UI runTeam never sends one), and the UI re-enables the button after ~1.5s reload. _Fix:_ Derive a default idempotencyKey (e.g. hash of contentTrackId+objective within a time window) in enqueueContentGraphJob or the route, and disable/debounce the UI button until the job is confirmed.
- ⚪ **data-integrity** — Packet provenance (sourceIdsUsed/insightIdsUsed) is taken from the RESEARCHER's cited indexes, but the copywriter is only handed the evidence point TEXT (no ids) and told to use 'ONLY' some of it. The final copy can drop or reshape claims while the packet still advertises the full researcher citation set. _Fix:_ Have the copywriter (or a post-step) tag which evidence points it actually used and intersect provenance to those; or attach provenance at the point level rather than the whole researcher output.

- ⚪ **scaling** — defaultGetTrack lists up to 200 active tracks and finds the target by id in memory (listContentTracks limit 200). _Fix:_ Fetch the single track by id (store.getTrackById) instead of listing-and-finding.

### jobs-workers

**Purpose:** A Postgres-backed job queue plus out-of-process worker runtime — the OS's async backbone. enqueueJob() inserts pending jobs (deduped by idempotency key via a partial unique index). processNextJob() atomically claims the next runnable job with FOR UPDATE SKIP LOCKED, runs its registered handler from generalRegistry, records completion/retry/failure with exponential backoff, appends a job_attempts row, emits audit events, and upserts a worker heartbeat. runWorker() is the poll loop; worker.ts is the `npm run worker` entrypoint polling the 'general' queue. A recently-added reaper (reclaimStalledJobs) resets jobs stranded 'active' by crashed workers. Pure orchestration — it runs handlers, it is not itself an LLM/agent module.

**Data tables:** jobs, job_attempts, worker_heartbeats, automation_rules (via automations enqueue path), audit_events (via writeAuditEvent)  
**External deps:** none  
**LLM calls per run:** 0  
**UX state:** WorkersPage in os-ui.tsx (~line 4267) exists and renders, but is thin and read-only: KPIs (online/stale/jobs/running), a worker-heartbeat list, and job-queue counts byStatus + byQueue. There is NO per-job list, no failure-reason surfacing, no retry/cancel/re-drive controls, and no dead-letter view. Operators can see that N jobs failed but not which, why, or how to recover them from the OS.

**Data flow:** Jobs ENTER via enqueueJob() from many callers: POST /api/jobs, the Ask router (only content.generate, which is 'available'), automations runAutomation/fireEventRules, and per-module enqueue helpers (content.generate, content.graph, library.import, publishing.dispatch, knowledge.compile, audit.paid, intelligence.scout/analyze/dream — ALL targeting queue 'general'). STORED in `jobs` (status pending). The single general worker process (worker.ts) claims via claimNext() UPDATE...SKIP LOCKED (increments attempts, sets locked_at, status active), looks up registry[type], runs the handler, then writes result + status completed to `jobs`, appends a `job_attempts` row, upserts `worker_heartbeats` (keyed heartbeat_<name>) and a file heartbeat, and emits job.completed/retry/failed audit events. On throw: evaluateJobFailure() requeues with exp backoff while attempts<maxAttempts, else marks 'failed' (terminal). Idle loop every ~30 cycles runs reclaimStalledJobs() to reset stale 'active' rows. Failed jobs FLOW nowhere further (no dead-letter, no auto-retry). The Workers page reads worker_heartbeats + a listJobs summary via /api/workers.

Its own comment claims it is 'called by the event bus / hooks' but no such wiring exists. _Fix:_ Call fireEventRules(eventType, metadata) from writeAuditEvent() (src/lib/audit/index.ts) after the event is persisted, or from a dedicated event-bus dispatch. - 🟠 **not-wired** — No scheduler exists for cadence/cron work. automation_rules supports triggerType='schedule' with a cron-ish `schedule` column, but nothing anywhere evaluates due schedule rules and calls runAutomation. There is also no cadence tick for the many functions documented as 'safe to run on a schedule' (memory sweep, source refresh, intelligence scout/dream). The only periodic mechanism in the whole system is the stalled-job reaper bound to worker idle cycles. _Fix:_ Add a scheduler entrypoint (e.g. src/workers/scheduler.ts as a separate `npm run scheduler` process, or a tick inside runWorker) that periodically loads enabled schedule rules, computes due ones from `schedule`+lastRunAt, and calls runAutomation. Persist next_run_at to avoid double-firing across workers.
- 🟠 **data-integrity** — The reaper can double-run legitimately long jobs. reclaimStalledJobs() resets any 'active' job with locked_at older than a fixed 5 min back to 'pending' purely on elapsed time, with no check of the owning worker's heartbeat/liveness. Multi-agent handlers (content.generate, content.graph, knowledge.compile, audit.paid — several sequential OpenRouter calls) can exceed 5 min. For maxAttempts=3 job types the reclaim requeues to 'pending' while the original is still running, so a second worker runs it concurrently. _Fix:_ Make liveness heartbeat-based, not time-based: only reclaim an 'active' job whose worker's worker_heartbeats row is stale (or store worker_id/lease_expires_at on the job and have handlers renew the lease). Raise/ër per-job-type the timeout above realistic handler duration, and make handlers idempotent.
- 🟡 **not-wired** — video-worker.ts is a stub. It writes only a JSON file heartbeat every 30s; it never polls a queue, never looks up a handler, and never writes to the worker_heartbeats table — so it does no work and never even appears on the Workers page. No 'video.*' job type is registered in generalRegistry either. _Fix:_ Either implement a real runWorker loop for a 'video' queue with registered video.* handlers and DB heartbeats, or remove the stub and its npm script until the pipeline exists.
- 🟡 **scaling** — Only the 'general' queue is ever consumed, but enqueue targets are unvalidated. All built-in job types correctly use queue 'general', so they run. However runAutomation enqueues with the rule's arbitrary actionQueue (default 'general' but user-settable to anything) and arbitrary actionType. A rule pointing at any queue other than 'general' enqueues a job no worker polls; a rule with an actionType absent from generalRegistry enqueues a job that can never succeed. _Fix:_ Validate actionType against knownJobTypes() and actionQueue against the set of queues that have a running worker when creating/enabling an automation rule; reject or warn in the API.
- 🟡 **reliability** — Reaper runs only on idle cycles (idleCycles % 30 === 0). A worker that is continuously busy never increments idleCycles, so it never runs the reaper. With effectively one general worker, if it crashes mid-job the job stays 'active' until that same worker restarts and happens to go idle — there is no independent, always-on reaper. _Fix:_ Run the reaper on a wall-clock interval independent of idle state (e.g. every N seconds regardless of throughput), or move it to the scheduler process so it runs even when all workers are saturated.
- 🟡 **error-handling** — No dead-letter queue and no failure observability. Terminal failures just set status='failed' with a failureReason column. There is no dead-letter table, no retry-from-failed path, no alerting, and the Workers UI surfaces only status counts — not a failed-job list or the failureReason text. _Fix:_ Add a failed-jobs list endpoint + UI panel showing type/failureReason/attempts and a re-enqueue action; consider a dead_letter flag or table and alert on failure-rate thresholds.
- ⚪ **efficiency** — Missing-handler jobs are retried instead of dead-lettered immediately. processNextJob() routes a 'no handler registered' condition through failJob()/evaluateJobFailure(), so with default maxAttempts=3 the job is requeued with exponential backoff twice before terminating — even though a missing handler is a permanent, non-transient error. _Fix:_ When registry[job.type] is absent, mark the job failed terminally in one shot (bypass retry) since it can never succeed until code is deployed.
- ⚪ **scaling** — Heartbeat rows are keyed by deterministic id `heartbeat_<workerName>` (one row per worker NAME). Two 'general' worker processes (horizontal scale) upsert the same row, clobbering each other's status/currentJobId. _Fix:_ Key heartbeats by a unique per-process id (hostname+pid) with workerName as a display field, so each running process gets its own row.
- ⚪ **ux** — Heartbeat is only refreshed between jobs / on idle, and UI staleness threshold (30s) is far tighter than the reaper threshold (5 min). A single job running longer than 30s makes the worker render as 'stale' on the Workers page despite being healthy. _Fix:_ Emit an in-progress heartbeat on a timer while a job runs (or reconcile the two thresholds), so 'live' reflects real process health during long handlers.

### n8n-handoff

**Purpose:** A signed webhook bridge between WOBBLE OS and n8n. Outbound: when a content packet is APPROVED and has PASSED quality review, it serializes the packet into a canonical JSON payload, HMAC-signs it (SHA-256 over `timestamp.body`), POSTs it to a registered webhook endpoint, and records the delivery as a webhook_event; failures are written to a dead_letters table and the packet's n8nHandoffStatus is flipped (queued -> sent/failed). Inbound: /api/n8n/callback receives HMAC-signed callbacks from n8n, verifies signature + replay window + idempotency, and logs them as inbound webhook_events. Two sibling public webhooks (zernio delivery status, intelligence ingestion) live in /api/webhooks and share the same HMAC/fail-closed pattern. There are NO LLM calls in this module — it is pure transport/security plumbing.

**Data tables:** webhook_endpoints (read), webhook_events (read/write), dead_letters (write-only — never read back), content_packets (read + update n8nHandoffStatus), audit_logs (write via writeAuditEvent), intelligence_items (write, via sibling /api/webhooks/intelligence -> ingestIntelligencePayload)  
**External deps:** n8n (outbound HTTP POST to registered endpoint URLs), zernio (sibling inbound webhook, post-status delivery), apify/intelligence (sibling inbound ingestion webhook)  
**LLM calls per run:** 0  
**UX state:** Thin, read-only tile with a status-labeling bug.

**Data flow:** OUTBOUND: caller POSTs {endpointId, contentPacketId, idempotencyKey, requestedBy, approvedBy} to /api/n8n/handoff/content -> Zod validate -> sendApprovedContentToN8n() checks webhook_events for a duplicate idempotencyKey -> loads endpoint + content_packet -> assertContentPacketReadyForHandoff (must be approved + quality passed + not already sent) -> resolves signing secret from process.env[endpoint.secretRefName] -> builds canonical payload (stableJson), signs HMAC-SHA256 over `timestamp.body` -> sets packet n8nHandoffStatus=queued -> POSTs to endpoint.url via fetch -> on 2xx: writes success webhook_event + packet=sent + audit; on error/non-2xx: writes failed webhook_event + dead_letters row + packet=failed + audit. INBOUND: n8n POSTs to /api/n8n/callback -> receiveN8nCallback() requires N8N_WEBHOOK_SECRET (fail-closed), verifies timestamp within 300s + HMAC signature, checks idempotencyKey against webhook_events, and logs an inbound webhook_event. The inbound callback performs NO state mutation beyond logging — it does not drive any content/publish status. GET /api/n8n returns endpoints + last 50 events for the Handoff UI tile.

**Flaws (10):**

- 🟠 **not-wired** — dead_letters rows are written on every failed outbound handoff but NOTHING ever reads or processes them. There is no retry worker, no scheduler, no recovery job, and no UI surface. Grep for deadLetters/retryCount across src shows only the writer (n8n/index.ts) and schema — no consumer. _Fix:_ Build a dead-letter worker (or wire into the existing job-queue) that selects dead_letters where status='open', re-invokes sendApprovedContentToN8n with the original idempotencyKey (now safe once idempotency is DB-enforced), increments retryCount with backoff, and flips status to resolved/archived. Add a dead-letter list + manual 'Retry' action to HandoffPage.
- 🟠 **data-integrity** — Idempotency is enforced only by an application-level read-then-insert (findWebhookEventByIdempotencyKey -> insertWebhookEvent). webhook_events.idempotency_key has NO unique index in schema.ts (contrast the jobs table at line 64 which DOES have a partial unique index). The lookup+insert is not transactional. _Fix:_ Add uniqueIndex on webhook_events.idempotency_key (partial: where idempotency_key is not null) and rely on a unique-violation catch to detect duplicates instead of a prior SELECT; wrap the send/insert in a transaction.
- 🟠 **data-integrity** — A single content packet can be handed off multiple times using DIFFERENT idempotency keys. The only guard against re-publish is assertContentPacketReadyForHandoff throwing when n8nHandoffStatus==='sent', but status is only set to 'sent' AFTER the transport returns (line 259). Between the read (line 155) and the 'sent' write, a second request with a fresh idempotencyKey passes the assert. _Fix:_ Make the duplicate guard keyed on contentPacketId (e.g. unique partial index on content_packets requiring a single in-flight handoff, or a DB-level status transition approved->sending guarded by an UPDATE ... WHERE n8nHandoffStatus IN ('pending','failed') RETURNING). Set the reservation before the transport, not after.
- 🟠 **security** — POST /api/n8n/handoff/content has NO authentication/authorization — no session check, no shared secret, no HMAC on the trigger itself. It trusts caller-supplied requestedBy and approvedBy strings as free text. _Fix:_ Require an authenticated founder session (or an internal service token) on the handoff trigger route, and derive requestedBy/approvedBy from the authenticated identity / the packet's actual approval record rather than trusting the request body.
- 🟡 **correctness** — Broken status classification. GET /api/n8n counts 'failed' as status NOT IN ('ok','delivered','received') and the UI colors an event green only for status IN ('ok','received','delivered'). But webhook_events.status is only ever one of 'pending'|'success'|'failed'|'duplicate'|'rejected' (see domain type line 30). None of 'ok'/'delivered'/'received' are ever written. _Fix:_ Align the string sets to the actual enum: treat 'success' (and 'duplicate','accepted') as healthy and 'failed'/'rejected' as failed, in both the route counts and the UI Tag color.
- 🟡 **not-wired** — receiveN8nCallback is a pure logger. On a valid, verified inbound callback it inserts a 'success' webhook_event and writes an audit event, but performs zero downstream action — it never parses the callback body to update a content packet's publish status, mark delivery, or trigger anything. _Fix:_ After verification, route by eventType to a handler that updates the referenced content packet / creates the corresponding record, inside the same idempotency guard.
- 🟡 **security** — replayProtected is set to true on inbound events even when idempotencyKey is null (lines 423-425). When the caller omits X-Wobble-Idempotency-Key, the only replay defense is the 300s timestamp window — there is no nonce store — so a captured valid signed request can be replayed unlimited times for 5 minutes, yet it is logged as replay-protected. _Fix:_ Either require idempotencyKey on inbound (reject when absent), or persist a nonce/(signature) dedupe within the window; set replayProtected to reflect whether a dedupe key was actually enforced.
- 🟡 **security** — Sibling public webhooks /api/webhooks/zernio and /api/webhooks/intelligence HMAC over the raw body only, with NO timestamp/replay window (unlike the n8n callback which signs `timestamp.body` and enforces a 300s window). _Fix:_ Adopt the same timestamped-signature + replay-window scheme (verifyWebhookSignature) for both sibling webhooks, or add a nonce/dedupe on ingestion.
- ⚪ **security** — The signing secret is resolved via process.env[endpoint.secretRefName] where secretRefName is an arbitrary DB-stored string, and GET /api/n8n exposes secretRefName (env var names) unauthenticated. _Fix:_ Whitelist allowed secret-ref names, gate GET /api/n8n behind auth, and redact secretRefName from the response.
- ⚪ **error-handling** — Missing N8N_WEBHOOK_SECRET on the inbound callback throws, which route.ts maps to HTTP 500 rather than a deliberate 503 'disabled' (as the intelligence/zernio siblings do). _Fix:_ Return a 503 with an explicit 'callback webhook disabled — set N8N_WEBHOOK_SECRET' message, matching the sibling webhooks and the fail-closed convention.

## 1) TOP 15 FLAWS OVERALL (ranked, cross-module)

**1. No scheduler / cron exists anywhere in the OS — every "cadence" subsystem is dead.**
- Module: architectural (memory-brain, research-radar, library-scheduler, source-registry)
- Severity: CRITICAL
- Issue: There is no interval worker, cron, or recurring-job mechanism. Handlers exist and are even registered, but nothing enqueues them on a cadence: `harvestPendingConversations` + `purgeExpiredArchivedMemory` (memory), `dispatchDuePosts`/`publishing.dispatch` (library), the `trend_radar` "schedule" agent (radar), and source intake runs.
- Impact: Chats never become memory; archived memory never purges (unbounded leak); non-manual scheduled posts never fire and become permanently stuck rows; radar/intelligence never refreshes on its own. The "team of agents that runs continuously" vision is, in production, a set of manual buttons.
- Fix: Introduce one scheduling primitive (n8n cron, a systemd timer, or an in-process interval worker) that enqueues the already-registered jobs, plus POST triggers for each. This single fix reactivates four modules.

**2. Source intake pipeline is non-functional out of the box — approving a source produces no scrape/transcript/chunks.**
- Module: source-registry (feeds learning-engine)
- Severity: CRITICAL
- Issue: `createSourceIntakeRun` only inserts a `source_intake_runs` row; nothing dequeues or executes it. The per-type `defaultConnectedAgents` (orchestrator, transcript/visual/competitor specialists) are string arrays never dispatched. Apify exists but is wired to pitch/intelligence, not here.
- Impact: The whole ingestion → knowledge chain is inert unless an external n8n worker manually POSTs chunks. Approving a source triggers a knowledge-compile job that finds zero chunks and skips. The advertised multi-agent intake is decorative.
- Fix: Build a real `source.intake` job + handler that dispatches by type to scraper/transcript/vision, writes chunks, and completes the run internally.

**3. Zernio (and all non-manual publishers) cannot fetch local media — auto-posting is broken for the entire seeded library.**
- Module: library-scheduler
- Severity: CRITICAL
- Issue: `zernioMediaItems()` hands Zernio a `/api/library/assets/{id}/media` URL, but `/api/library/*` is not in `proxy.ts` PUBLIC_PREFIXES, so the edge gate 401s the (cookie-less) provider.
- Impact: Every Zernio publish/schedule of a locally-backed asset (i.e. all imported ad/reel folders) fails or posts broken media. The headline "flip-the-switch auto-posting" upgrade does not work for real content.
- Fix: Add a dedicated signed, short-lived public media endpoint (HMAC token in URL) or pre-signed object-storage URLs — never make the session route public wholesale.

**4. Unauthenticated writes bypass the founder trust gate and poison downstream knowledge.**
- Module: source-registry (blast radius: learning-engine → every agent)
- Severity: HIGH (effectively critical given blast radius)
- Issue: `POST /api/sources`, `/intake`, `PATCH …/intake/[runId]`, and `/chunks` have no `requireFounder` and no HMAC. Anyone reachable can PATCH a run to `succeeded` with arbitrary `extractedData`/`confidence`/`cost` and POST arbitrary chunks onto an approved source.
- Impact: The approve-before-use guarantee is void for everything downstream of approval; injected chunks flow into the Knowledge Compiler and become "trusted" retrievable notes served to all agents.
- Fix: `requireFounder` on create/intake/complete/chunks; verify a shared-secret HMAC on n8n callbacks.

**5. Memory retrieval has no default bank isolation — personal/client memory leaks to any query.**
- Module: memory-brain
- Severity: HIGH
- Issue: `retrieveMemoryCandidates` filters by bank only if `bankSlugs` is passed; `retrieveMemoryContext` and Ask WOBBLE call it with none, so one query semantically searches every bank (personal founder banks, confidential client/project memory, `rejected_ideas`).
- Impact: Cross-scope data leak — a founder's personal notes or client-confidential facts surface to any caller/context. Combined with the unauthenticated `/api/memory/retrieve` route (flaw #10 territory), unauthenticated cross-bank reads are possible.
- Fix: Require an explicit permitted-bank allow-list at retrieval; default-exclude personal/client banks; thread caller identity through.

**6. Knowledge Compiler prompt-injection poisons the single retrieval contract every agent reads.**
- Module: learning-engine (blast radius: strategy, copywriting, SEO, research, ask)
- Severity: HIGH
- Issue: Raw chunk content is concatenated verbatim as `[i] <content>` into the compiler prompt with no delimiting/hardening. Output notes are embedded, interlinked, and served via the one `retrieveKnowledge()` contract.
- Impact: A hostile scraped page or crafted `/chunks` payload can make the compiler emit attacker-controlled "notes" that propagate silently to every downstream agent. Broadest single blast radius in the OS.
- Fix: Fence chunk text as untrusted data-not-instructions; strip instruction patterns; consider a post-parse trust check before persisting.

**7. Ask WOBBLE's entire agentic orchestrator + tool registry is unreachable from the UI.**
- Module: ask-wobble
- Severity: CRITICAL (feature-dead, not a data bug)
- Issue: `AskPage.send()` only calls `/api/ask` and `/api/ai/chat`; zero frontend callers of `/api/ask/agent`, and no confirm button to send `confirmActions=true`.
- Impact: All of Ask WOBBLE's "do stuff" power — system inspection, propose/apply model swap, remember/forget/pin memory, and the destructive-action confirmation gate — is dead shipped code.
- Fix: Wire AskPage to `/api/ask/agent` (or fall through for action intents) and add the pending-confirmation UI.

**8. Ingested source chunks are never embedded — all raw source data is permanently invisible to retrieval.**
- Module: source-registry (retrieval-gap into learning-engine)
- Severity: HIGH
- Issue: `buildSourceChunkRows` hardcodes `embedding:null`, `attachSourceChunks` inserts as-is, but `searchSourceChunks` filters `isNotNull(embedding)`.
- Impact: The "raw source fidelity" half of the knowledge contract returns nothing; downstream agents lose all verbatim grounding — only compiler notes survive, and only if the compile ran.
- Fix: Embed chunks at attach time; add a backfill script.

**9. Unbounded input-token budget in Ask WOBBLE (100% of Q&A traffic).**
- Module: ask-wobble
- Severity: HIGH
- Issue: Every question injects the FULL untruncated content of all core+active `memory_records`, plus ~8 memory chunks, 8×3 source chunks, the whole system snapshot, and 12 intel items — with no total token cap.
- Impact: Cost per question grows without bound as the Brain grows; large Brains push trivial questions into tens of thousands of input tokens and eventually context-limit 500s.
- Fix: Cap/truncate the brain block and enforce a total evidence-token budget before the call.

**10. Hot-path tables have no indexes; the most-hit route triggers full scans.**
- Module: command-center + memory-brain
- Severity: HIGH
- Issue: `model_runs` (highest-volume table, one row per LLM/search/media call) has no indexes, yet `costSummary()` runs three full-table `SUM` scans on every home-dashboard load. `approvals` has no `status` index behind a `COUNT WHERE status='pending'` on every load. `memory_chunks.embedding` has no ANN (hnsw/ivfflat) index, so every retrieval and every dedup-on-write is a sequential scan.
- Impact: The home page and all memory retrieval degrade linearly with data volume; DB load compounds.
- Fix: Add `model_runs(created_at)`, `approvals(status, created_at)`, a pgvector hnsw index on `memory_chunks.embedding`, plus btree indexes on memory status/tier.

**11. Zernio scheduling creates orphaned and mis-routed un-cancellable posts.**
- Module: library-scheduler
- Severity: HIGH
- Issue: `schedulePost()` pushes to Zernio BEFORE inserting the local row (crash → orphan with no `publisher_ref`, webhook can never match). Separately, `scheduleRemote` is chosen by "is Zernio configured," not by the post's publisher — so `publisher='ayrshare'/'n8n'` posts get pushed to Zernio, and `killRemote` only deletes when `publisher==='zernio'`, so they can never be cancelled.
- Impact: Ghost posts go live with zero local tracking; mis-routed posts fire on the wrong account and are un-cancellable. Brand/money risk.
- Fix: Insert local row first (`status='scheduling'`), then push remotely, then persist ref with a compensating delete on failure; make `scheduleRemote` publisher-specific; restrict the publisher enum to publishers that have an adapter and a firing path.

**12. Content graph has no checkpointing — a flaky node re-pays the whole pipeline (2 Sonnet nodes) for zero output.**
- Module: content-command
- Severity: HIGH
- Issue: Any of nodes 1/2/3/5 returning unparseable JSON throws and fails the job, discarding all prior billed calls; with `maxAttempts:2` the full 5-call pipeline re-runs from scratch.
- Impact: Transient formatting failures waste up to ~8–10 model calls (including two strong-tier) to produce nothing.
- Fix: Persist node outputs and resume from the failed node; add one bounded "return STRICT JSON only" re-ask before throwing.

**13. Notes/memory written without an embedder are permanently unretrievable, and retrieval fakes a 0.75 score.**
- Module: learning-engine + memory-brain
- Severity: HIGH
- Issue: When no embedder is configured, notes insert with `embedding=null` and `searchNotes` filters `isNotNull` → never retrievable, no dedup, no links, no backfill path. Memory's recency fallback stamps a hardcoded `0.75` similarity on every row, and dedup/conflict silently disables itself.
- Impact: A compile/write run before the key is set "succeeds" but rots silently; retrieval degrades to recency with fabricated confidence and no signal to the founder.
- Fix: Refuse to persist null-embedding notes (fail → retry), add a backfill job, and mark fallback results as non-semantic instead of emitting a fake score.

**14. Destructive-action confirmation is per-turn and non-pinned (TOCTOU).**
- Module: ask-wobble
- Severity: HIGH
- Issue: `confirmActions:true` authorises every `requiresConfirmation` tool for the whole turn, and on re-call the model regenerates from scratch rather than replaying the exact tool+args the founder saw.
- Impact: Confirming "apply upgrade X" can, on the re-run, apply a different/additional irreversible action (`apply_model_upgrade`, `forget_memory`) — what gets applied isn't necessarily what was shown, and one confirm green-lights unlimited destructive calls.
- Fix: Pin confirmation to the specific pending tool-call id+args; scope per-action, not per-turn. (Also: the `remember` tool is not gated at all — auto-saves durable facts on a single model turn.)

**15. "Research Radar" does no retrieval — it presents hallucinated market signals with confident 0–100 scores.**
- Module: research-radar
- Severity: HIGH
- Issue: Despite being a "market/competitor signal scan," `generateRadarScan` makes no web/search/RSS call. Signals come purely from LLM training data + the founders' own approved intel. The registered `market_researcher`/`trend_radar` agents (declared search_api/rss tools) never run, and the `radar_scout` role isn't even seeded (falls back to gpt-4o-mini).
- Impact: Founders make decisions on plausible but potentially stale/invented "competitor moves" and "tech releases." A trust-eroding, high-visibility failure.
- Fix: Wire a real retrieval step (Apify/SERP/RSS) before the LLM, or relabel output as "hypotheses to verify" in prompt + UI; seed the role to a strong model.

## 2) ARCHITECTURAL GAPS THAT SPAN MODULES

**A. No scheduler → all cadence/agent-team behavior is dead.** (memory harvester + purge, library dispatch, radar trend, source intake). This is the single highest-leverage architectural fix. See flaw #1.

**B. "Registered but never executed" agents everywhere.** `agents.ts` registers `market_researcher`, `trend_radar` (radar) and each source-type's `source_intake_orchestrator` + specialists, but no dispatcher ever invokes them; Ask WOBBLE's whole tool registry is unreachable. Capability/system-map reporting overstates what actually runs. Either implement the dispatch paths or delete the decorative registrations so the system map is honest.

**C. Retrieval is silently broken in two places.** Source chunks are never embedded (invisible), and any note/memory written without an embedder is permanently unretrievable. The retrieval layer treats "no embedding" as "empty result" rather than an error, so ingestion can look successful while producing zero retrievable knowledge. Add a hard invariant: no row that is meant to be retrievable may persist with `embedding=null` — fail or backfill.

**D. Uniform prompt-injection exposure at every ingestion boundary.** Ask, memory, radar, source-registry, learning-engine, and content all concatenate untrusted scraped/ingested text verbatim into prompts. The Knowledge Compiler is the worst because its output feeds the single `retrieveKnowledge` contract every agent reads — one poisoned source poisons the whole OS. Needs a shared "untrusted-data fence + data-not-instructions" convention applied at every retrieval-into-prompt site, with the do-not-say/grounding rules always last.

**E. Broken isolation / no auth on read+write routes.** No `middleware.ts` exists. Costs/audit/approvals/agents (command-center), memory retrieve/list (memory), knowledge retrieve/notes (learning), and source create/intake/chunks (source-registry) are all unauthenticated. Combined with memory's no-default-bank-scoping, an unauthenticated caller can read the full financials, audit trail, knowledge base, and cross-founder/client memory, and write poisoned sources. A single auth layer (session for founders, HMAC service token for n8n) is missing and needed OS-wide.

**F. Cost/token blowups are systemic, not local.** Ask (full brain), content-graph (full untruncated notes/chunks, no idempotency → double-runs on double-click), source/knowledge (`z.array(string).min(1)` with no `.max`, up to 40–60 unbounded chunks into one prompt), radar (truncation → parse-crash). There is no shared "context budget" primitive; every module reinvents (or omits) it. A per-prompt token budget + per-chunk char cap should be a shared util.

**G. Non-atomic multi-step writes across modules.** Memory (`approveMemoryUpdate`, `createMemoryRecord`, `deleteRecordCascade`), library (Zernio push before local insert), and learning-engine (`reinforceNote` read-modify-write → lost updates under concurrency). None use `db.transaction`/atomic SQL. Partial failures leave orphans, consumed-approval-with-no-memory, or double-published posts.

**H. Fabricated/placeholder telemetry undermines the command surface.** The Command Center ships two hardcoded KPI tiles ("9 of 26 modules", "17 CI passing") already contradicted by the handoff log, and today's spend can silently undercount (0-cost estimates). The dashboard founders use to watch the OS presents fiction next to real numbers.

**I. Inconsistent quality gates.** The "premium" multi-agent content graph gates on the scoring LLM's self-report only, while the cheaper single-call fallback uses an objective `gradeContentExcellence`. The stronger path has the weaker gate — self-graded homework auto-passes into the approval queue.

**J. Media generation doesn't exist.** Media Studio is a FAL_KEY status page with no fal client and no generation path, despite being presented as the video/image studio — an entire advertised capability is absent.

## 3) ONE-LINE HEALTH VERDICT PER MODULE

- **command-center** — MINOR GAPS: works and handles states cleanly, but ships fabricated KPIs, no auth, no indexes on the hottest route, and no refresh.
- **ask-wobble** — NEEDS WORK: two surfaces polished, but the entire agent/tool surface is unreachable, tokens are unbounded, confirmation is unsafe (TOCTOU), and evidence is injection-exposed.
- **memory-brain** — NEEDS WORK: rich UI and CRUD, but no default bank isolation (cross-scope leak), harvester/purge unwired, no ANN index, and non-transactional writes.
- **research-radar** — NEEDS WORK: polished UI over a backend that does zero retrieval (hallucinated "signals"), decorative agents, unseeded role, and parse-crash fragility.
- **source-registry** — NEEDS WORK (arguably worst): intake is inert, chunks never embedded, and unauthenticated writes bypass the trust gate that the whole knowledge base depends on.
- **learning-engine** — NEEDS WORK: strong compile-not-summarize design, but injection poisons the one contract every agent reads, null-embedding notes rot silently, and retrieve/notes routes are open.
- **content-command** — MINOR GAPS: genuinely deep and functional; real issues are no checkpointing (wasted spend), self-graded gate, unbounded context, and no idempotency.
- **library-scheduler** — NEEDS WORK: polished UI, but Zernio can't fetch local media (auto-posting broken), the dispatch loop is dead, and scheduling creates orphaned/mis-routed un-cancellable posts.
- **media-studio** — STUB / NEEDS BUILD: status page only; no fal client, no image/video generation despite being presented as the generation studio.

---

**Single highest-ROI fixes, in order:** (1) add one scheduler and wire the already-registered jobs; (2) add a shared auth/HMAC layer + default memory bank scoping; (3) enforce the "no retrievable row without an embedding" invariant and embed source chunks; (4) apply one untrusted-data fence at every ingest→prompt boundary; (5) fix Zernio media access + insert-before-remote ordering to make publishing real.
